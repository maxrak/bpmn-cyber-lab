Anti-Pattern: Message Events e Token Appeso

Boundary event fuori tempo: il messaggio di annullamento arriva quando il task a cui è agganciato è già completato

← Torna all'indice della documentazione

📨 Il problema: un messaggio che non trova più il destinatario

Il Boundary Message Event è un evento di confine che ascolta un messaggio mentre il task a cui è agganciato è in esecuzione. Non appena il messaggio arriva, il task viene interrotto e il flusso si devia. Ma cosa succede se quel messaggio arriva dopo che il task è già stato completato? Il messaggio non trova il suo destinatario, viene ignorato — e il processo rimane bloccato in uno stato di attesa infinita.

⚠️ Scenario di questo anti-pattern

Cliente e Venditore si scambiano messaggi per gestire un ordine. Il Venditore usa un boundary event agganciato a Task_ProcessOrder per intercettare un eventuale annullamento da parte del Cliente.

Il problema: entro il tempo in cui il Cliente rivede l'ordine e decide di annullarlo, il Venditore ha già completato Task_ProcessOrder e il token si è spostato su Event_ReceivePayment. Il boundary event non è più attivo, il messaggio di annullamento viene perso, e il token del Venditore rimane sospeso ad aspettare un pagamento che non arriverà mai.

💡 Come usare questo esempio
  1. Carica anti-pattern-message-events-example.bpmn dal menu File
  2. Avvia la Simulation
  3. Segui la sequenza descritta nella sezione "Come simularlo" — in particolare, completa prima Task_ProcessOrder nel pool Venditore, poi annulla nel pool Cliente
  4. Osserva il token del Venditore che rimane bloccato su "Attendi Pagamento"

🗺️ Struttura del processo

Il diagramma mostra due pool che collaborano via message flow:

Pool: Cliente

Start → Prepara Ordine → Invia Richiesta ──msg──▶ → Attendi Conferma ◀──msg── → Rivedi Ordine → [XOR: Accetta?] │ Sì → Invia Pagamento ──msg──▶ │ → Attendi Spedizione │ → Ricevi Merce → End ✅ │ └ No → Annulla Ordine ──msg──▶ → End (Annullato)

Pool: Venditore

Start (msg) → Valida Ordine → Verifica Disponibilità → Invia Conferma ──msg──▶ → Task_ProcessOrder │ ├─ [completato] → Event_ReceivePayment ⏳ │ → Verifica Pagamento │ → Prepara Spedizione │ → Notifica Spedizione ──msg──▶ │ → Spedisci Merce → End ✅ │ └─ Boundary_CancelOrder ◀──msg── (solo se task attivo!) → Annulla Ordine → End

PROBLEMA Il boundary event ha una finestra temporale limitata

Il Boundary_CancelOrder è un interrupting boundary message event agganciato a Task_ProcessOrder. Questo significa che può intercettare il messaggio solo mentre Task_ProcessOrder è in esecuzione.

Nel simulatore bpmn-js, i task completano non appena l'utente clicca su di essi. Non appena il Venditore completa Task_ProcessOrder, il token avanza su Event_ReceivePayment — e il boundary event cessa di essere in ascolto.

Se in quel momento (o dopo) il Cliente invia il messaggio di annullamento, non esiste più nessun receiver nel processo del Venditore. Il token del Venditore rimarrà bloccato su Event_ReceivePayment per sempre, aspettando un pagamento che il Cliente non invierà mai.

🧪 Come simulare il token appeso

Cliente
Avvia il processo

Clicca su Start nel pool Cliente. Completa Prepara Ordine e poi Invia Richiesta: il messaggio parte verso il Venditore e istanzia il suo processo.

Venditore
Elabora l'ordine ricevuto

Nel pool Venditore compaiono i task attivi. Completa in sequenza: Valida Ordine → Verifica Disponibilità → Invia Conferma → Task_ProcessOrder.

⚠️ Momento critico
Task_ProcessOrder viene completato

Non appena clicchi su Task_ProcessOrder per completarlo, il token avanza automaticamente su Event_ReceivePayment. Il boundary event Boundary_CancelOrder viene disattivato — non è più in ascolto di nessun messaggio.

Cliente
Riceve conferma e decide di annullare

Nel pool Cliente, completa Attendi Conferma → Rivedi Ordine. Al gateway "Accetta?" scegli il ramo No: viene eseguito Annulla Ordine che invia il messaggio Message_CancelOrder.

🔴 Token appeso
Il messaggio arriva, ma non trova destinatario

Il messaggio Message_CancelOrder non viene ricevuto da nessuno nel pool Venditore. Il token del Venditore rimane bloccato su Event_ReceivePayment, in attesa di un pagamento che non arriverà mai. Il processo non si chiude.

💡 Verifica visiva nel simulatore
  • Il pool Cliente termina regolarmente su "Ordine Annullato" ✅
  • Il pool Venditore mostra il token fermo su "Attendi Pagamento" — evidenziato nel simulatore come task in attesa (cerchio con busta + bordo azzurro)
  • Non c'è modo di proseguire: nessun click avanza il processo
  • L'unica uscita è riavviare la simulazione

🔬 Analisi tecnica

SBAGLIATO Boundary event agganciato al task sbagliato

Il Boundary_CancelOrder è definito come boundary event su Task_ProcessOrder. La semantica BPMN è chiara: un boundary event è attivo only while its host task is active.

<!-- ❌ ANTI-PATTERN: boundary event su Task_ProcessOrder --> <task id="Task_ProcessOrder" name="Elabora Ordine"> <incoming>Flow_S4</incoming> <outgoing>Flow_S5</outgoing> </task> <!-- Questo boundary event smette di funzionare non appena Task_ProcessOrder viene completato --> <boundaryEvent id="Boundary_CancelOrder" name="Annullamento" attachedToRef="Task_ProcessOrder" cancelActivity="true"> <messageEventDefinition messageRef="Message_CancelOrder" /> </boundaryEvent> <!-- Dopo il task, il token va qui... --> <intermediateCatchEvent id="Event_ReceivePayment" name="Attendi Pagamento"> <messageEventDefinition messageRef="Message_PaymentInfo" /> </intermediateCatchEvent> <!-- ...ma a questo punto il boundary event è GIÀ MORTO. Se arriva Message_CancelOrder ora, viene IGNORATO. -->

Il messageFlow di collegamento punta a Boundary_CancelOrder:

<!-- Flow_Msg_5 punta al boundary event: --> <messageFlow id="Flow_Msg_5" sourceRef="Event_SendCancel" targetRef="Boundary_CancelOrder" /> <!-- Quando arriva, Boundary_CancelOrder non è più attivo → il messaggio viene perso → token appeso -->

Perché questo accade nel simulatore bpmn-js

In un motore BPMN reale (es. Camunda, Flowable), i task possono rimanere attivi per ore o giorni in attesa di input umano. Durante quell'attesa, il boundary event è in ascolto. In bpmn-js la simulazione è manuale e sincrona: un task diventa "completato" non appena l'utente ci clicca sopra. Non esiste parallelismo asincrono reale tra i due pool.

Il risultato pratico è che l'ordine in cui l'utente clicca determina se il boundary event fa o meno in tempo ad intercettare il messaggio:

Ordine di click Risultato
Venditore: completa Task_ProcessOrder prima che il Cliente annulli 🔴 Token appeso su Event_ReceivePayment
Cliente: annulla mentre Task_ProcessOrder è ancora attivo (prima di completarlo) ✅ Il boundary event intercetta — flusso corretto

Un processo corretto non dovrebbe dipendere dall'ordine di esecuzione. L'anti-pattern produce un comportamento non deterministico: si ottengono risultati diversi a seconda di quando si clicca.

✅ La soluzione: Event-Based Gateway

La correzione consiste nel sostituire il boundary event con un Event-Based Gateway posizionato dopo Task_ProcessOrder. Il gateway rimane attivo in attesa di uno qualunque dei messaggi previsti; il primo che arriva attiva il ramo corrispondente e disabilita tutti gli altri.

Questa è la struttura adottata in message-events-example.bpmn (la versione corretta).

CORRETTO Event-Based Gateway dopo Task_ProcessOrder

<!-- ✅ CORRETTO: Event-Based Gateway come punto di attesa --> <task id="Task_ProcessOrder" name="Elabora Ordine"> <incoming>Flow_S4</incoming> <outgoing>Flow_S5</outgoing> </task> <!-- Il gateway aspetta uno dei due messaggi alternativi --> <eventBasedGateway id="Gateway_WaitForDecision" name="Pagamento o Annull.?"> <incoming>Flow_S5</incoming> <outgoing>Flow_S5b</outgoing> <!-- → Event_ReceivePayment --> <outgoing>Flow_S5c</outgoing> <!-- → Event_ReceiveCancel --> </eventBasedGateway> <!-- Ramo 1: arriva il pagamento → prosegui normalmente --> <intermediateCatchEvent id="Event_ReceivePayment" name="Attendi Pagamento"> <messageEventDefinition messageRef="Message_PaymentInfo" /> </intermediateCatchEvent> <!-- Ramo 2: arriva l'annullamento → gestisci cancellazione --> <intermediateCatchEvent id="Event_ReceiveCancel" name="Annullamento"> <messageEventDefinition messageRef="Message_CancelOrder" /> </intermediateCatchEvent> <!-- Message flow: punta all'intermediate catch event, non a un boundary --> <messageFlow id="Flow_Msg_5" sourceRef="Event_SendCancel" targetRef="Event_ReceiveCancel" />

Come funziona il flusso corretto

Task_ProcessOrder [completato] ↓ Gateway_WaitForDecision (eventBasedGateway) ├── Event_ReceivePayment ◀──msg── Cliente invia pagamento │ → Task_VerifyPayment → ... → End (Ordine Evaso) ✅ │ └── Event_ReceiveCancel ◀──msg── Cliente annulla ordine → Task_CancelOrder → End (Ordine Annullato) ✅

SBAGLIATO Boundary Event

  • Attivo solo durante il task ospite
  • Il messaggio arriva dopo: ignorato
  • Token appeso su Event_ReceivePayment
  • Processo non si chiude mai
  • Comportamento non deterministico

CORRETTO Event-Based Gateway

  • Rimane attivo fino all'arrivo del primo messaggio
  • Pagamento o annullamento: attende entrambi
  • Mutuamente esclusivo: una sola risposta viene processata
  • Processo sempre convergente
  • Comportamento deterministic e testabile

📋 Regole pratiche per i Message Event

  1. Usa un Boundary Message Event solo per interruzioni che possono realisticamente avvenire mentre il task è ancora in esecuzione. Se il task si completa istantaneamente (o quasi), la finestra di ascolto è troppo breve per essere affidabile.
  2. Quando il processo deve attendere uno tra più messaggi alternativi, usa un Event-Based Gateway posizionato dopo l'ultimo task sincronizzato. Il gateway resta in ascolto di tutti i messaggi configurati finché uno non arriva.
  3. Non puntare mai un message flow a un boundary event se il messaggio può arrivare dopo la fine del task. In quel caso, converte il boundary event in un intermediate catch event raggiungibile dall'Event-Based Gateway.
  4. Verifica sempre che ogni token abbia un percorso verso un End Event in tutti gli scenari possibili. Un token appeso è spesso il sintomo di un message event che non ha trovato il suo destinatario.
  5. In fase di simulazione: testa sempre lo scenario "peggiore" (annullamento tardivo). Completa prima tutti i task del Venditore fino al punto di attesa, poi simula l'annullamento dal pool Cliente.
