Esempio: Message Events

Comunicazione asincrona tra processi tramite eventi messaggio

← Torna all'indice della documentazione

💬 Obiettivo dell'esempio

Il file message-events-example.bpmn dimostra come utilizzare i Message Events in BPMN per modellare la comunicazione asincrona tra processi indipendenti. Questo esempio mostra una collaborazione tra Cliente e Venditore in un tipico flusso di ordine e-commerce, illustrando tutti i principali tipi di message eventi.

💡 Perché usare Message Events?

I Message Events sono fondamentali quando modelli processi che coinvolgono più partecipanti (persone, sistemi, organizzazioni) che comunicano in modo asincrono. A differenza di una semplice chiamata di funzione, un messaggio:

  • Non blocca il processo mittente (comunicazione asincrona)
  • Può attraversare confini di sistema (es: API, email, messaggi)
  • Rappresenta interazioni reali tra entità indipendenti
  • Può essere perso, ritardato o gestito con timeout
📖 Come utilizzare questo esempio
  1. Carica message-events-example.bpmn dal menu File
  2. Osserva nel Modeler i due pool (Cliente e Venditore) e i message flow che li collegano
  3. Passa alla Simulation per vedere come i messaggi sincronizzano i due processi
  4. Nota come ogni messaggio ha un mittente (throw event) e un ricevente (catch event)

🛒 Scenario: Processo di ordine e-commerce

L'esempio modella un processo completo di acquisto online che coinvolge due partecipanti indipendenti:

👤 Pool Cliente

Responsabilità:

  • Preparare e inviare la richiesta d'ordine
  • Attendere conferma dal venditore
  • Decidere se accettare o rifiutare
  • Inviare pagamento (se accetta)
  • Ricevere notifica di spedizione
  • Opzione: annullare in qualsiasi momento

🏢 Pool Venditore

Responsabilità:

  • Ricevere richiesta d'ordine (avvia processo)
  • Validare ordine e verificare disponibilità
  • Inviare conferma al cliente
  • Elaborare ordine (interrompibile)
  • Ricevere e verificare pagamento
  • Preparare e spedire merce
  • Gestire eventuali annullamenti
Flusso messaggi:

Cliente: Prepara ordine → [Msg 1: Richiesta Ordine] → Venditore: Riceve e valida
Venditore: Conferma → [Msg 2: Conferma Ordine] → Cliente: Rivedi e decidi
Cliente: Pagamento → [Msg 3: Info Pagamento] → Venditore: Verifica e prepara
Venditore: Spedizione → [Msg 4: Notifica Spedizione] → Cliente: Ricevi merce
(Opzionale) Cliente: Annulla → [Msg 5: Annulla Ordine] → Venditore: Interrompi (Boundary Event)

📨 Tipi di Message Events

📤Message Intermediate Throw Event

Simbolo: Cerchio doppio con icona busta (vuoto)

Funzione: Inviare un messaggio a un altro partecipante

Comportamento: Il processo emette il messaggio e continua immediatamente senza aspettare risposta. È un'operazione non bloccante.

Nel diagramma di esempio:
  • Event_SendOrder - Cliente invia richiesta d'ordine
  • Event_SendConfirmation - Venditore invia conferma
  • Event_SendPayment - Cliente invia dati pagamento
  • Event_SendShipment - Venditore notifica spedizione
  • Event_SendCancel - Cliente annulla ordine

📥Message Intermediate Catch Event

Simbolo: Cerchio doppio con icona busta (pieno)

Funzione: Attendere la ricezione di un messaggio specifico

Comportamento: Il processo si ferma in attesa del messaggio. Quando il messaggio arriva, il token prosegue.

Nel diagramma di esempio:
  • Event_ReceiveConfirmation - Cliente aspetta conferma dal venditore
  • Event_ReceivePayment - Venditore aspetta pagamento dal cliente
  • Event_ReceiveShipment - Cliente aspetta notifica di spedizione

Nota: Questi sono punti di sincronizzazione tra i due processi. Un processo non può proseguire finché non riceve il messaggio atteso.

▶️Message Start Event

Simbolo: Cerchio singolo con icona busta (pieno)

Funzione: Avviare un processo quando arriva un messaggio specifico

Comportamento: Il processo rimane inattivo finché non riceve il messaggio che lo attiva. Questo è tipico di processi "reattivi" o "event-driven".

Nel diagramma di esempio:
  • Event_ReceiveOrder - Il processo del Venditore inizia solo quando riceve la richiesta d'ordine dal cliente

Caso d'uso: Perfetto per modellare sistemi che reagiscono a eventi esterni (API endpoint, email in arrivo, messaggi da code).

⚠️Message Boundary Event

Simbolo: Cerchio doppio con icona busta, attaccato al bordo di un'attività

Funzione: Interrompere un'attività in corso quando arriva un messaggio

Comportamento: Mentre l'attività è in esecuzione, se arriva il messaggio specificato, l'attività viene interrotta e il flusso segue il percorso del boundary event.

Nel diagramma di esempio:
  • Boundary_CancelOrder - Attaccato a Task_ProcessOrder nel processo Venditore
  • Se il cliente invia un messaggio di annullamento mentre il venditore sta elaborando l'ordine, l'elaborazione viene interrotta immediatamente
  • Il flusso passa a Task_CancelOrder per gestire l'annullamento

Caso d'uso: Gestione eccezioni, timeout, richieste di cancellazione, escalation.

🔗 Message Flow vs Sequence Flow

È fondamentale comprendere la differenza tra i due tipi di connettori in BPMN:

→ Sequence Flow (freccia solida)
  • All'interno di un singolo processo/pool
  • Rappresenta il flusso di controllo
  • Indica l'ordine di esecuzione delle attività
  • Sincrono - passa il token direttamente
Esempio: Task A → Task B significa "esegui B dopo A"
⤏ Message Flow (freccia tratteggiata)
  • Tra due processi/pool diversi
  • Rappresenta lo scambio di messaggi
  • Collega throw event a catch event
  • Asincrono - comunicazione non bloccante
Esempio: Pool A [invia] ⤏ Pool B [riceve] significa "A invia messaggio a B"
💡 Regola importante

Non puoi usare sequence flow tra pool diversi. La comunicazione tra partecipanti indipendenti deve sempre avvenire tramite message flow. Questo riflette la realtà: processi indipendenti non condividono il flusso di esecuzione, ma comunicano tramite messaggi.

🎮 Guida alla simulazione

Fase 1: Avvio del processo

1. Il processo Cliente inizia automaticamente con lo Start Event
2. Esegue "Prepara Ordine"
3. Raggiunge "Invia Richiesta" (Throw Event)
4. → Message Flow attivato →
5. Il processo Venditore si AVVIA con "Ricevi Richiesta" (Start Event)
6. Venditore esegue "Valida Ordine" e "Verifica Disponibilità"

Osserva: Il processo Venditore non esisteva prima di ricevere il messaggio. Il Message Start Event lo ha attivato. Questo è chiamato pattern "Request-Response".

Fase 2: Conferma e decisione

7. Venditore raggiunge "Invia Conferma" (Throw Event)
8. → Message Flow →
9. Cliente si SBLOCCA da "Attendi Conferma" (Catch Event)
10. Cliente esegue "Rivedi Ordine"
11. Cliente decide al gateway XOR: Accetta? (Sì/No)

Osserva: Il Cliente era bloccato in attesa della conferma. I Catch Event sono punti di sincronizzazione. Questo garantisce che il Cliente non proceda prima di avere la risposta del Venditore.

Fase 3a: Percorso di successo (Cliente accetta)

12a. Cliente → Gateway: Sì → "Invia Pagamento" (Throw Event)
13a. → Message Flow →
14a. Venditore si SBLOCCA da "Attendi Pagamento" (Catch Event)
15a. Venditore: "Verifica Pagamento" → "Prepara Spedizione" → "Notifica Spedizione"
16a. → Message Flow →
17a. Cliente si SBLOCCA da "Attendi Spedizione" (Catch Event)
18a. Cliente: "Ricevi Merce" → End Event ✅
19a. Venditore: "Spedisci Merce" → End Event ✅

Fase 3b: Percorso di annullamento (Cliente rifiuta)

12b. Cliente → Gateway: No → "Annulla Ordine" (Throw Event)
13b. → Message Flow →
14b. Venditore: "Elabora Ordine" viene INTERROTTO dal Boundary Event
15b. Venditore esegue "Annulla Ordine" → End Event ❌
16b. Cliente → End Event (Annullato) ❌

Osserva: Il Boundary Event permette al messaggio di annullamento di interrompere un'attività in corso. Questo è essenziale per gestire eccezioni asincrone.

🎯 Suggerimenti per la simulazione
  • Usa la vista Pools: Assicurati che entrambi i pool siano visibili per vedere la sincronizzazione
  • Velocità lenta: Imposta velocità bassa per osservare i message flow
  • Pausa strategica: Metti in pausa quando raggiungi i Catch Event per vedere il "waiting"
  • Log eventi: Controlla il log per vedere l'ordine cronologico di invio/ricezione messaggi
  • Testa entrambi i percorsi: Simula con "Accetta" e con "Rifiuta" per vedere i due scenari

🌍 Applicazioni nel mondo reale

🏦 Sistemi finanziari
  • Richiesta → Approvazione → Pagamento
  • Message flow tra cliente, banca, merchant
  • Boundary events per timeout o cancellazioni
📧 Workflow email
  • Invio email = Throw Event
  • Attesa risposta = Catch Event
  • Reminder = Boundary Timer
🔧 Microservizi
  • API call = Message Flow
  • Service A → Message → Service B
  • Event-driven architecture
💼 Esempio pratico: Sistema di ticketing
Pool Cliente:
Start → Apri Ticket (Throw: "Nuova Richiesta") → [Attendi Risposta] (Catch) → Valuta Soluzione
→ Gateway: Risolto? → Sì: Chiudi Ticket | No: Rilancia (Throw: "Aggiornamento")

Pool Support:
[Ricevi Richiesta] (Start) → Analizza → Throw: "Prima Risposta" → [Lavora Ticket]...
Boundary: [Escalation] se passa troppo tempo → Passa al Manager

🎓 Conclusioni chiave

Quando usare Message Events

  • Collaborazione tra pool/partecipanti diversi: Mai usare sequence flow tra pool
  • Comunicazione asincrona: Il mittente non aspetta risposta immediata
  • Sincronizzazione inter-processo: Catch Events creano punti di attesa
  • Eventi esterni che avviano processi: Message Start Event per sistemi event-driven
  • Interruzioni asincrone: Boundary Events per eccezioni, timeout, cancellazioni
⚠️ Errori comuni da evitare
  • ❌ Usare sequence flow tra pool diversi (usa message flow)
  • ❌ Dimenticare di collegare throw e catch event con message flow
  • ❌ Non associare correttamente i message definitions agli eventi
  • ❌ Aspettarsi comportamento sincrono da message event (sono asincroni)
  • ❌ Usare message event quando una semplice task sarebbe sufficiente (overhead inutile)

Best Practices

  • ✅ Nomina i messaggi in modo descrittivo ("Conferma Ordine" non "Messaggio 1")
  • ✅ Usa Message Start Events per processi che devono essere "triggerati" dall'esterno
  • ✅ Considera sempre timeout e gestione eccezioni (Boundary Events)
  • ✅ Documenta il payload del messaggio (cosa contiene il messaggio)
  • ✅ Usa pools separati per entità realmente indipendenti
