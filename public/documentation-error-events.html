🚨 Error Events - Gestione Errori

Impara a gestire errori ed eccezioni nei processi BPMN

📋 Obiettivo dell'Esempio

Questo esempio dimostra come utilizzare gli Error Events in BPMN per gestire errori ed eccezioni durante l'esecuzione di un processo. Gli Error Events permettono di implementare strategie di error handling, recovery e compensazione in modo strutturato e visuale.

Concetto Chiave: Error Events

Gli Error Events rappresentano situazioni eccezionali che interrompono il flusso normale di un processo. A differenza dei Message Events (comunicazione) e Signal Events (broadcast), gli Error Events sono specificamente progettati per la gestione delle eccezioni e il recovery da errori.

🎯 Cos'è un Error Event?

Un Error Event è un evento BPMN che permette di modellare e gestire condizioni di errore all'interno di un processo. Gli errori possono essere lanciati (thrown) e catturati (caught) in punti specifici del processo.

Caratteristiche degli Error Events:

  • Interrupting: Gli Error Boundary Events interrompono l'attività a cui sono attaccati
  • Propagazione: Gli errori si propagano verso l'alto fino al primo handler disponibile
  • Tipizzazione: Ogni errore ha un nome e un codice univoco per identificarlo
  • Scope limitato: Gli errori sono visibili solo all'interno del processo/subprocess
  • Recovery: Permettono di implementare logiche di compensazione e retry

🛒 Scenario: Order Processing System

L'esempio simula un sistema di processamento ordini e-commerce che può fallire in diversi punti. Il processo include tre fasi critiche, ciascuna con possibili errori:

💳 Payment Processing

Possibile errore: Carta di credito rifiutata

Recovery: Notifica cliente, suggerisci metodi alternativi

PaymentError

📦 Inventory Check

Possibile errore: Prodotto esaurito

Recovery: Rimborso pagamento, notifica cliente

InventoryError

🚚 Shipping Arrangement

Possibile errore: Corriere non disponibile

Recovery: Rimborso, rilascia stock, notifica cliente

ShippingError

📝 Definizioni degli Errori

Il processo definisce quattro tipi di errori, ciascuno con un nome e un codice univoco:

PaymentError
Codice: ERR_PAYMENT

Errore durante il processo di pagamento (carta rifiutata, fondi insufficienti)

InventoryError
Codice: ERR_INVENTORY

Prodotto non disponibile in magazzino

ShippingError
Codice: ERR_SHIPPING

Impossibile organizzare la spedizione (corriere non disponibile)

SystemError
Codice: ERR_SYSTEM

Errore di sistema generico (gestito da Event Subprocess)

🔧 Tipi di Error Events

❌

Error End Event

Termina un processo o subprocess lanciando un errore che può essere catturato da un Boundary Event.

Nell'esempio:

  • End_PaymentError - Lancia PaymentError se il pagamento fallisce
  • End_InventoryError - Lancia InventoryError se lo stock non è disponibile
  • End_ShippingError - Lancia ShippingError se il corriere non è disponibile
🛡️

Error Boundary Event

Cattura errori che si verificano durante l'esecuzione di un'attività o subprocess, interrompendo l'attività in corso.

Nell'esempio:

  • Boundary_PaymentError - Cattura errori dal subprocess di pagamento
  • Boundary_InventoryError - Cattura errori dal subprocess di inventario
  • Boundary_ShippingError - Cattura errori dal subprocess di spedizione
🔄

Error Start Event (in Event Subprocess)

Avvia un subprocess dedicato alla gestione degli errori quando un errore specifico viene lanciato. Può essere interrupting o non-interrupting.

Nell'esempio:

  • Start_SystemError - Gestione centralizzata degli errori di sistema (non-interrupting)
  • Log dell'errore e notifica al supporto IT
  • Il processo principale continua l'esecuzione
⚡

Propagazione degli Errori

Gli errori si propagano verso l'alto attraverso i livelli di subprocess fino a trovare un handler (Boundary Event o Error Start Event) appropriato.

Comportamento:

  • Se non catturato, l'errore termina l'intero processo
  • Gli Error Events sono specifici per tipo (nome/codice)
  • Un handler può catturare un errore specifico o generico

♻️ Strategie di Compensazione

Ogni error handler implementa una strategia di compensazione appropriata al punto del processo in cui si verifica l'errore:

Errore Fase del Processo Azioni di Compensazione Esito
PaymentError Inizio (nessuna risorsa allocata) 1. Notifica cliente
2. Suggerisci metodo alternativo 		Cancellazione ordine
InventoryError Dopo pagamento (pagamento completato) 1. Rimborso pagamento
2. Notifica cliente out of stock 		Cancellazione ordine
ShippingError Dopo inventory (pagamento + stock riservato) 1. Rimborso pagamento
2. Rilascio stock riservato
3. Notifica cliente problema spedizione 		Cancellazione ordine
SystemError In qualsiasi momento 1. Log errore
2. Notifica supporto IT
3. Processo continua (non-interrupting) 		Monitoraggio parallelo

Principio della Compensazione

Più avanti nel processo si verifica un errore, più azioni di compensazione sono necessarie per "annullare" le operazioni già completate. Questo è visibile nella sequenza crescente di compensazioni: PaymentError (solo notifica) → InventoryError (+ rimborso) → ShippingError (+ rimborso + rilascio stock).

  • Recovery da problemi di inventario e disponibilità prodotti

  • Compensazione automatica per operazioni parzialmente completate

🏥 Healthcare

  • Gestione errori nei processi di prenotazione e scheduling

🚀 Pattern Avanzati

1. Error Event Subprocess (Non-Interrupting)

L'esempio include un Event Subprocess con Error Start Event non-interrupting per gestire errori di sistema senza fermare il processo principale:

Caratteristiche:

Non-Interrupting: Il processo principale continua mentre viene eseguita la gestione dell'errore in parallelo.

Logging e Monitoring: Utile per registrare errori e allertare il team IT senza bloccare l'operatività.

Use Case: Errori non critici che richiedono attenzione ma non devono fermare il flusso.

2. Cascading Error Handling

Gli errori possono essere catturati a diversi livelli di un processo complesso:

  • Livello Task: Errori specifici di una singola attività
  • Livello Subprocess: Errori all'interno di un subprocess (come nell'esempio)
  • Livello Process: Errori globali catturati a livello di processo principale

3. Retry Pattern

Sebbene non implementato nell'esempio, è possibile estendere il pattern per includerel'attività di retry:

Implementazione Retry:

  • Cattura l'errore con un Boundary Event
  • Aggiungi un gateway per verificare il numero di tentativi
  • Reindirizza al task originale se sotto il limite di retry
  • Procedi con la compensazione se superato il limite

4. Error Escalation

Per errori che richiedono intervento umano o decisioni manageriali:

  • Cattura l'errore e crea un task per il team appropriato
  • Usa Timer Events per escalation automatica se non gestito entro un timeout
  • Integra con sistemi di ticketing o notifica

💡 Best Practices

✅ Quando Usare Error Events

  • Condizioni eccezionali: Situazioni che non fanno parte del flusso normale
  • Errori tecnici: Problemi di connettività, timeout, system failures
  • Validazioni fallite: Dati non validi, business rules violate
  • Risorse non disponibili: Stock esaurito, servizi offline
  • Transazioni fallite: Pagamenti rifiutati, operazioni non completate

Quando NON Usare Error Events

Flussi decisionali normali: Usa Exclusive Gateway invece di Error Events per decisioni business standard (es: "Cliente VIP?" → Yes/No).

Validazioni previste: Se una condizione è prevedibile e frequente (es: età < 18), modellala come parte del flusso normale, non come errore.

🎯 Design Guidelines

  • Nomi descrittivi: Usa nomi chiari per error types (PaymentError, InventoryError)
  • Codici univoci: Assegna error codes univoci per facilitare debugging e logging
  • Granularità appropriata: Non creare troppi error types, ma nemmeno troppo pochi
  • Documentazione: Documenta cosa causa ogni errore e come viene gestito
  • Compensazione completa: Assicurati che ogni error handler annulli tutte le operazioni necessarie
  • Logging: Registra sempre gli errori per analisi e debugging

⚙️ Implementation Tips

  • Interrupting vs Non-Interrupting: Scegli basandoti sulla criticità dell'errore
  • Error Propagation: Struttura subprocess nidificati per error handling granulare
  • Timeout handling: Combina Error Events con Timer Events per gestire timeout
  • User notification: Sempre notificare gli utenti quando un processo fallisce
  • Audit trail: Mantieni traccia di tutti gli errori e le azioni di recovery

📊 Confronto: Error vs Message vs Signal

Caratteristica Error Events Message Events Signal Events
Scopo Gestione eccezioni Comunicazione punto-a-punto Broadcast a processi multipli
Visibilità Solo all'interno del processo/subprocess Tra processi specifici Globale (tutti i processi)
Direzione Verso l'alto (propagazione) Sender → Receiver specifico Broadcaster → N listeners
Comportamento Interrupting (tipicamente) Varia (può essere interrupting o no) Varia (può essere interrupting o no)
Tipizzazione Tipizzato (nome + codice) Tipizzato (nome messaggio) Tipizzato (nome segnale)
Use Case Error handling, recovery, compensazione Interazione tra pools, richieste/risposte Allarmi globali, eventi broadcast
Accoppiamento Stretto (padre-figlio) Medio (sender-receiver) Loose (emitter non conosce receivers)
