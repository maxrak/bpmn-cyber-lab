Esempio: Confronto Gateway

Analisi comparativa dei diversi tipi di gateway BPMN

← Torna all'indice della documentazione

📊 Obiettivo dell'esempio

Il file gateway-comparison.bpmn è stato progettato per illustrare visivamente il comportamento dei tre principali tipi di gateway BPMN utilizzati per controllare il flusso di esecuzione in un processo. Caricando questo esempio nel Modeler e simulandolo nel modulo Simulation, potrai osservare le differenze pratiche nel modo in cui i token attraversano ciascun tipo di gateway.

💡 Come utilizzare questo esempio
  1. Apri il file gateway-comparison.bpmn dal menu File nel Modeler
  2. Passa al modulo Simulation (il file verrà caricato automaticamente)
  3. Avvia la simulazione e osserva come i token si dividono e si ricongiungono
  4. Presta particolare attenzione ai punti di split e join di ciascun gateway

🏗️ Struttura del diagramma

Il processo è organizzato in modo da confrontare direttamente i tre gateway:

Flusso generale:

Start Event → Parallel Gateway (Split) →
    → Path 1: Exclusive Gateway (XOR) →
    → Path 2: Inclusive Gateway (OR) →
Parallel Gateway (Join) → End Event

Il Parallel Gateway iniziale divide il flusso in due percorsi paralleli, permettendo di osservare contemporaneamente il comportamento dell'Exclusive e dell'Inclusive Gateway. Alla fine, un altro Parallel Gateway sincronizza i due percorsi prima di terminare.

🔀 Tipi di Gateway

⬥Exclusive Gateway (XOR)

Simbolo: Rombo vuoto o con una "X"

Comportamento (Split): Il token segue uno e uno solo dei percorsi in uscita, basandosi su condizioni o sul primo percorso disponibile.

Comportamento (Join): Attende un token da uno qualsiasi dei percorsi in ingresso, poi prosegue immediatamente.

Nel diagramma di esempio: L'Exclusive Gateway divide il flusso in 3 opzioni (Task User 1, Task Service 1, Task Manual 1), ma solo una verrà eseguita. Il gateway di join aspetta il completamento di quel singolo percorso.

⬥Parallel Gateway (AND)

Simbolo: Rombo con un "+"

Comportamento (Split): Il token viene duplicato e inviato contemporaneamente su tutti i percorsi in uscita.

Comportamento (Join): Attende che arrivino token da tutti i percorsi in ingresso prima di proseguire (sincronizzazione).

Nel diagramma di esempio: Il primo Parallel Gateway crea due token simultanei che eseguono in parallelo il path XOR e il path OR. Il secondo Parallel Gateway aspetta che entrambi i percorsi siano completati prima di proseguire verso l'End Event.

⬥Inclusive Gateway (OR)

Simbolo: Rombo con un cerchio interno

Comportamento (Split): Il token viene inviato su uno o più percorsi in uscita, basandosi su condizioni. Se più condizioni sono vere, vengono eseguiti tutti i percorsi corrispondenti.

Comportamento (Join): Attende token da tutti i percorsi attivi (cioè quelli che sono stati effettivamente attivati durante lo split).

Nel diagramma di esempio: L'Inclusive Gateway divide il flusso in 3 opzioni (Task Service 2, Task Manual 2, Task Send 2), e una o più possono essere eseguite in base alle condizioni. Il gateway di join attende il completamento di tutti i percorsi attivati.

📋 Tabella comparativa

Caratteristica Exclusive (XOR) Parallel (AND) Inclusive (OR)
Percorsi attivati (Split) Esattamente 1 Tutti 1 o più (in base a condizioni)
Decisione Basata su condizioni mutualmente esclusive Nessuna decisione, sempre tutti Basata su condizioni indipendenti
Sincronizzazione (Join) Primo token in arrivo Tutti i token da tutti i percorsi Tutti i token dai percorsi attivi

Esempio business Approvazione: SI o NO Preparazione fattura + Spedizione merce Notifiche: email, SMS, push (una o più)

🎯 Osservazioni durante la simulazione

1. Avvio del processo

Quando avvii la simulazione, vedrai un token partire dallo Start Event e raggiungere il primo Parallel Gateway. Qui il token si divide in due token che procedono simultaneamente:

  • Un token entra nel percorso con l'Exclusive Gateway
  • Un token entra nel percorso con l'Inclusive Gateway

2. Exclusive Gateway in azione

Osserva attentamente quando il token raggiunge l'Exclusive Gateway (Split):

  • Il token sceglie una sola delle tre task disponibili
  • Le altre due task rimangono non eseguite
  • Al gateway di join, il processo prosegue immediatamente dopo aver completato la singola task

3. Inclusive Gateway in azione

Parallelamente, osserva il comportamento dell'Inclusive Gateway:

  • Il token può attivare una, due o tutte e tre le task
  • Se più task vengono attivate, vengono eseguite in parallelo
  • Al gateway di join, il processo attende il completamento di tutte le task attivate

4. Sincronizzazione finale

Il Parallel Gateway (Join) finale aspetta che:

  • Il percorso dell'Exclusive Gateway sia completato (1 task)
  • Il percorso dell'Inclusive Gateway sia completato (1-3 task)

Solo quando entrambi i percorsi hanno terminato, il token finale procede verso l'End Event.

💡 Suggerimento per la simulazione
  • Usa la velocità di simulazione lenta per osservare meglio il comportamento dei token
  • Metti in pausa la simulazione quando i token raggiungono i gateway per analizzare le scelte
  • Osserva il log degli eventi per vedere l'ordine di esecuzione delle attività
  • Riavvia la simulazione più volte: potresti vedere comportamenti diversi nell'Inclusive Gateway

🎓 Conclusioni chiave

Quando usare ciascun gateway

  • Exclusive Gateway: Quando devi scegliere esattamente una tra diverse alternative (es: approvazione SI/NO, routing basato su categoria)
  • Parallel Gateway: Quando devi eseguire tutte le attività contemporaneamente e sincronizzare alla fine (es: attività indipendenti ma necessarie)
  • Inclusive Gateway: Quando devi scegliere una o più alternative in base a condizioni non esclusive (es: notifiche multiple, controlli paralleli opzionali)
⚠️ Nota: Complex Gateway

Esiste anche un quarto tipo di gateway chiamato Complex Gateway, utilizzato per logiche di merge/split molto complesse con regole personalizzate. Tuttavia, non è supportato dalla libreria di simulazione token utilizzata in BPMN Cyber Lab, quindi non è incluso in questo esempio.
