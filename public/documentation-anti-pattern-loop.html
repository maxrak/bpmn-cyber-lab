Anti-Pattern: Loop senza Condizione di Uscita

Un processo che riprova all'infinito — e non termina mai

← Torna all'indice della documentazione

🔁 Il problema: retry senza limite

I loop di retry sono un pattern legittimo e comune in BPMN: se un'operazione fallisce, si riprova. Il problema nasce quando il loop non ha una condizione di uscita esplicita — nessun contatore, nessun timer, nessun numero massimo di tentativi. In questo caso, se la condizione di successo non viene mai soddisfatta, il token entra in un ciclo infinito che non converge mai verso un End Event.

⚠️ Perché questo anti-pattern è insidioso
  • Il diagramma sembra corretto a prima vista: c'è un gateway con due uscite, una va avanti e una riprova
  • Il problema è invisibile finché la condizione di successo non viene raggiunta — il modello non mostra la casistica di fallimento continuo
  • In produzione, un processo bloccato in un loop infinito consuma risorse e non può essere ripreso normalmente
  • In un contesto di sicurezza, un attaccante che controlla le condizioni di ingresso potrebbe forzare il loop per esaurire le risorse (Denial of Process)
💡 Come usare questo esempio
  1. Carica anti-pattern-loop.bpmn dal menu File
  2. Avvia la Simulation sul pool in alto (Anti-Pattern)
  3. Al gateway in rosso, scegli sempre il ramo "No / Riprova": il token torna al task iniziale
  4. Ripeti: puoi continuare a scegliere "No" senza limite — il processo non termina mai
  5. Poi apri il pool in basso (Corretto) e segui la stessa sequenza: dopo 3 "No" il processo si chiude su un End Event di errore

🔴 Anti-Pattern — Loop infinito

Scenario: il sistema tenta di elaborare una richiesta. Se l'elaborazione non va a buon fine, riprova. Non esistono limite di tentativi né percorso alternativo di uscita.

Start → Elabora Richiesta → [XOR: Elaborazione OK?] │ Sì ────────────┘──────────────→ End ✅ │ No / Riprova ──┘ │ └──→ Elabora Richiesta (di nuovo) │ [loop infinito ♻️]

SBAGLIATO XOR gateway senza uscita di errore

Il gateway ha solo due uscite: "Sì" (processo completato) e "No" (riprova). Il ramo "No" riporta direttamente al task precedente senza alcun controllo sul numero di iterazioni. Se "Sì" non viene mai scelto, il token cicla per sempre.

<!-- ❌ ANTI-PATTERN: XOR senza limite di tentativi --> <exclusiveGateway id="GW1" name="Elaborazione OK?"> <incoming>F1b</incoming> <outgoing>F1c</outgoing> <!-- Sì → End --> <outgoing>F1d</outgoing> <!-- No → torna al task --> </exclusiveGateway> <!-- Il ramo "No" loop back direttamente — nessun contatore, nessun limite, nessun exit --> <sequenceFlow id="F1d" name="No / Riprova" sourceRef="GW1" targetRef="T1_Elabora" />

Visibile nel simulatore: ogni volta che si sceglie "No / Riprova" al gateway rosso, il token ritorna a "Elabora Richiesta". Si può cliccare decine di volte — il processo non converge mai.

Come simulare il loop infinito

  1. Clicca Start nel pool Anti-Pattern
  2. Completa Elabora Richiesta
  3. Al gateway rosso, scegli "No / Riprova": il token torna al task
  4. Completa Elabora Richiesta di nuovo
  5. Scegli nuovo "No / Riprova" — puoi ripetere questo ciclo illimitatamente
  6. Il processo non ha modo di terminare finché non scegli "Sì"
  7. Nota: in un processo reale le condizioni potrebbero non permettere mai "Sì" (es. sistema downstream irraggiungibile)

✅ La soluzione: Loop con contatore e End Event di errore

Il pattern corretto aggiunge tre elementi: un task per incrementare un contatore, un secondo gateway che verifica se il limite è stato raggiunto, e un End Event di errore esplicito come uscita alternativa.

Start → Elabora Richiesta → [XOR: Elaborazione OK?] │ Sì ────────────┘──────────────→ End ✅ │ No ────────────┘ │ ↓ Incrementa Contatore Tentativi │ ↓ [XOR: Tentativi < 3?] │ │ Sì No │ │ │ ↓ │ End (Tentativi Esauriti ✖) │ └──→ Elabora Richiesta (loop limitato)

CORRETTO Contatore + gateway di limite + End Event di errore

Il secondo gateway (in verde nel diagramma) verifica il valore del contatore prima di ritornare al task. Quando il limite viene raggiunto, il flusso devia verso un End Event di errore esplicito — garantendo che il processo termini sempre, sia in caso di successo che di fallimento ripetuto.

<!-- ✅ CORRETTO: Aggiungere contatore + limite + exit --> <!-- Dopo il gateway "Elaborazione OK?" ramo No: --> <task id="T2_Counter" name="Incrementa Contatore Tentativi"> <incoming>F2d</incoming> <outgoing>F2e</outgoing> </task> <!-- Gateway di controllo limite (verde nel diagramma) --> <exclusiveGateway id="GW2b" name="Tentativi &lt; 3?"> <incoming>F2e</incoming> <outgoing>F2f</outgoing> <!-- No → esci con errore --> <outgoing>F2g</outgoing> <!-- Sì → riprova --> </exclusiveGateway> <!-- End Event di errore esplicito: il processo converge sempre --> <endEvent id="E2_Error" name="Tentativi Esauriti"> <incoming>F2f</incoming> <errorEventDefinition /> </endEvent>

Come simulare il loop con limite

  1. Clicca Start nel pool Corretto (pool in basso)
  2. Completa Elabora Richiesta → al primo gateway scegli No
  3. Completa Incrementa Contatore Tentativi → al gateway verde scegli Sì (tentativi < 3)
  4. Ripeti ancora due volte (2° e 3° tentativo): ogni volta il contatore aumenta
  5. Al 3° fallimento: il gateway verde riceve "Tentativi ≥ 3" → scegli No
  6. Il token raggiunge "Tentativi Esauriti" (End Event con simbolo ✖) — processo chiuso
  7. Alternativa: in qualsiasi tentativo scegli Sì al primo gateway → End "Richiesta Completata"

🛠️ Varianti del pattern di loop corretto

Il contatore numerico non è l'unico modo per garantire la convergenza di un loop. Ecco le alternative comuni:

Meccanismo di uscita Quando usarlo Elemento BPMN
Contatore numerico Numero max di tentativi noto (es. 3 retry) Task + exclusiveGateway su contatore
Timer boundary event Timeout assoluto sul task (es. "entro 30 minuti") timerBoundaryEvent sul task in loop
Error boundary event Il sistema downstream lancia un'eccezione al Nth fallimento errorBoundaryEvent sul task
Nessuna uscita Mai — ogni loop deve convergere ∅ — Anti-Pattern

VARIANTE Timer Boundary Event

Aggiungere un timer boundary sul task in loop garantisce che, indipendentemente dal numero di esecuzioni, il processo termina dopo un timeout:

<task id="T_Elabora" name="Elabora Richiesta"> <incoming>F_loop</incoming> <outgoing>F_next</outgoing> </task> <!-- Boundary: se il task dura più di 5 minuti, esci con timeout --> <boundaryEvent attachedToRef="T_Elabora" cancelActivity="true"> <timerEventDefinition> <timeDuration>PT5M</timeDuration> </timerEventDefinition> </boundaryEvent>

💡 Nota sul simulatore bpmn-js

Nel simulatore bpmn-js il timer boundary event non scatta automaticamente — il tempo simulato non avanza. Per testare il timeout è necessario cliccare manualmente sull'evento boundary.

Il contatore numerico (come in anti-pattern-loop.bpmn) è quindi la variante più adatta a una dimostrazione interattiva nel simulatore: l'utente sceglie esplicitamente il ramo "No" e la convergenza è visibile dopo il 3° tentativo.

📋 Regole pratiche per i loop in BPMN

  1. Ogni loop deve avere almeno un'uscita garantita. Identifica la condizione che impedisce il successo e modella esplicitamente il percorso alternativo verso un End Event (anche di errore).
  2. Usa un task "Incrementa Contatore" prima del gateway di limite. Questo rende il contatore visibile nel diagramma — aumenta la leggibilità e documenta automaticamente il meccanismo di retry.
  3. Etichetta sempre i rami del gateway di limite con valori chiari. "Tentativi < 3" / "Tentativi ≥ 3" è inequivocabile. "Sì / No" senza nome sul gateway è un anti-pattern di leggibilità.
  4. Usa un End Event di errore (simbolo ✖) per indicare il fallimento definitivo. Un None End Event su percorso di errore è ambiguo: sembra un successo. errorEventDefinition comunica chiaramente che il processo è terminato in uno stato anomalo.
  5. Valuta se un Timer Boundary Event è più appropriato di un contatore. Per operazioni di integrazione con sistemi esterni, un timeout assoluto (es. 30 minuti) è spesso più robusto di un numero fisso di retry — protegge anche da loop lenti.
