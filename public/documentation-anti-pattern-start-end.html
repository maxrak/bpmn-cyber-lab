Anti-Pattern: Start/End Events Errati

Errori comuni nella definizione di eventi di inizio e fine processo — e come correggerli

← Torna all'indice della documentazione

🚫 Cosa sono gli Anti-Pattern BPMN?

Un anti-pattern è una soluzione apparentemente ragionevole a un problema ricorrente che in realtà introduce ambiguità, errori logici o problemi di manutenibilità. In BPMN, gli anti-pattern più frequenti riguardano proprio gli eventi di start e di end, i confini semantici del processo.

⚠️ Perché gli anti-pattern degli eventi sono pericolosi
  • Il motore BPMN non sa quando istanziare il processo né quando liberare le risorse
  • Il team di analisi non può capire le condizioni di avvio e i possibili esiti
  • La documentazione generata automaticamente dal diagramma è inutilizzabile
  • In contesti di sicurezza e compliance, l'ambiguità può generare audit trail errati
💡 Come usare questo esempio
  1. Carica anti-pattern-start-end.bpmn dal menu File
  2. Osserva i tre pool: ciascuno illustra un anti-pattern distinto (elementi in rosso/arancio)
  3. Nota le annotazioni testuali collegate agli elementi problematici
  4. Passa alla Simulation per vedere il comportamento anomalo in azione

🔴 Anti-Pattern A — Start Events Multipli Ambigui

SBAGLIATO Due None Start Event identici

Due o più Start Event di tipo None (cerchio semplice) convergono sullo stesso task senza alcuna distinzione basata su condizione, messaggio o timer. Il diagramma non fornisce alcuna informazione su quando usare l'uno e quando l'altro.

<!-- ❌ SBAGLIATO: due None Start Event identici --> <startEvent id="Start_Manuale" name="Avvio Manuale"> <outgoing>Flow_a</outgoing> </startEvent> <startEvent id="Start_Auto" name="Avvio Automatico"> <outgoing>Flow_b</outgoing> </startEvent> <!-- Entrambi puntano allo stesso task: il lettore non sa quale scegliere --> <task id="Task_Elabora" name="Elabora Richiesta"> <incoming>Flow_a</incoming> <incoming>Flow_b</incoming> </task>

Problema concreto durante la simulazione: entrambi gli start event risultano cliccabili. Attivarne uno produce un token; attivarne entrambi produce due token concorrenti sullo stesso task — comportamento mai voluto e non documentato.

Come simularlo

  1. Apri il pool "Anti-Pattern A" nella Simulation
  2. Clicca su "Avvio Manuale": parte un token
  3. Riavvia e clicca su "Avvio Automatico": parte un altro token identico
  4. Oppure clicca su entrambi senza riavviare: due token indipendenti sullo stesso processo — impossibile nella realtà

CORRETTO Opzione 1: Tipi diversi

Differenziare il tipo semantico di ciascun start event:

<!-- ✅ None: avvio manuale --> <startEvent name="Richiesta Manuale" /> <!-- ✅ Message: email da cliente --> <startEvent name="Email Ricevuta"> <messageEventDefinition messageRef="Msg_Email" /> </startEvent> <!-- ✅ Timer: ogni lunedì mattina --> <startEvent name="Schedulato"> <timerEventDefinition> <timeCycle>0 9 * * 1</timeCycle> </timerEventDefinition> </startEvent>

CORRETTO Opzione 2: Start unico + Gateway

Un solo start event, poi un gateway discrimina il percorso:

<!-- ✅ Un solo punto di ingresso --> <startEvent name="Ricevi Richiesta" /> <exclusiveGateway name="Tipo di avvio?" /> <!-- Uscite etichettate chiaramente --> <sequenceFlow name="Manuale" sourceRef="GW" targetRef="T_Manual" /> <sequenceFlow name="Automatico" sourceRef="GW" targetRef="T_Auto" />

🟠 Anti-Pattern B — Processo Senza End Event

SBAGLIATO Il flusso termina su un Task

Il flusso di sequenza si chiude su un Task privo di outgoing, senza un End Event esplicito. Formalmente il BPMN è parzialmente valido (un task senza outgoing è un implicit end), ma è ambiguo e contro le linee guida BPMN 2.0.

<!-- ❌ SBAGLIATO: il processo "muore" dentro un task --> <startEvent id="Start" name="Avvia" /> <task id="T1" name="Ricevi Input"> <incoming>F1</incoming> <outgoing>F2</outgoing> </task> <task id="T2" name="Elabora Dati"> <incoming>F2</incoming> <outgoing>F3</outgoing> </task> <task id="T3" name="Salva Risultato"> <incoming>F3</incoming> <!-- nessun outgoing = nessun End Event --> </task>

Problema concreto: il motore BPMN mantiene l'istanza del processo in stato "running" indefinitamente. Nella simulazione, il token si ferma su "Salva Risultato" senza mai concludersi — impossibile verificare se il processo è terminato.

Come simularlo

  1. Apri il pool "Anti-Pattern B" nella Simulation
  2. Avvia il processo: il token percorre tutti e tre i task
  3. Dopo "Salva Risultato" il token rimane lì — nessuna fine
  4. Il processo resulta ancora "attivo" nel simulatore: risorse non liberate

CORRETTO Terminare sempre con End Event

Aggiungere sempre un End Event esplicito e, se ci sono esiti diversi, usare End Event con nomi significativi:

<!-- ✅ Fine esplicita --> <task id="T3" name="Salva Risultato"> <incoming>F3</incoming> <outgoing>F4</outgoing> </task> <endEvent id="End" name="Dati Salvati con Successo" /> <!-- Se può fallire, End Event con Error --> <endEvent id="End_Err" name="Errore Salvataggio"> <errorEventDefinition errorRef="Err_Save" /> </endEvent>

🔴 Anti-Pattern C — End Events Indistinguibili

SBAGLIATO Gateway senza condizioni, End Events identici

Un Exclusive Gateway non ha nome e le sue uscite non hanno etichette. Entrambi i rami terminano su End Event con lo stesso nome generico "Fine". È impossibile ricostruire la logica del processo solo dal diagramma.

<!-- ❌ SBAGLIATO: gateway vuoto, flussi senza nome, end identici --> <exclusiveGateway id="GW" name=""> <outgoing>Flow_c</outgoing> <outgoing>Flow_d</outgoing> </exclusiveGateway> <!-- uscite senza name/condizione --> <sequenceFlow id="Flow_c" sourceRef="GW" targetRef="End_A" /> <sequenceFlow id="Flow_d" sourceRef="GW" targetRef="End_B" /> <endEvent id="End_A" name="Fine" /> <!-- ❓ quale esito? --> <endEvent id="End_B" name="Fine" /> <!-- ❓ quale esito? -->

Problema concreto: nella simulazione il token arriva al gateway e il simulatore chiede "quale ramo seguire?" — ma le opzioni mostrate sono entrambe senza nome. Il modellatore non sa cosa scegliere.

Come simularlo

  1. Apri il pool "Anti-Pattern C" nella Simulation
  2. Avvia il processo e completa il task "Valuta Richiesta"
  3. Il simulatore mostra il gateway con due opzioni di uscita anonime
  4. Qualsiasi choice porti a un End Event chiamato "Fine": impossibile distinguere il risultato

CORRETTO Nominare tutto: gateway, flussi, end event

<!-- ✅ Gateway con nome chiaro --> <exclusiveGateway id="GW" name="Esito Valutazione?" /> <!-- Flussi etichettati con la condizione --> <sequenceFlow id="Flow_c" name="Approvata" sourceRef="GW" targetRef="End_OK" /> <sequenceFlow id="Flow_d" name="Rifiutata" sourceRef="GW" targetRef="End_KO" /> <!-- End Event con esiti distinguibili --> <endEvent id="End_OK" name="Richiesta Approvata" /> <endEvent id="End_KO" name="Richiesta Rifiutata" />

✅ Regole d'Oro per Start/End Events

  1. Ogni processo ha esattamente uno start event — oppure più start event di tipo semanticamente diverso (None, Message, Timer, Signal) ciascuno con uno scopo documentato.
  2. Ogni percorso di flusso termina con un End Event — nessun task deve essere l'ultimo elemento di un flusso senza un end event esplicito.
  3. Ogni End Event ha un nome che descrive l'esito — non "Fine" ma "Ordine Evaso", "Richiesta Rifiutata", "Errore Pagamento", ecc.
  4. Ogni Exclusive Gateway ha un nome che definisce la condizione — es. "Credito Sufficiente?", "Tipo Richiesta?".
  5. Ogni uscita di gateway ha un'etichetta con la condizione — es. "Sì / No", "Approvato / Rifiutato / Da Rivedere".
  6. Usa End Event tipizzati quando appropriato — Error End, Message End, Terminate End per comunicare chiaramente la natura della terminazione.
🎯 Test di autodiagnosi per i tuoi diagrammi

Fai queste domande ad ogni processo che modelli:

  • "Se ho due start event, so spiegare in una frase la differenza tra i due?"
  • "Ogni ramo del mio processo arriva a un End Event?"
  • "Guardando solo il nome degli End Event, capisco quali sono i possibili esiti del processo?"
  • "Le uscite dei miei gateway hanno etichette comprensibili anche per chi non conosce il dominio?"
