Esempio: Signal Events Multipli in BPMN

Gestione di Eventi Multipli

📡 Signal Events Multipli in BPMN

Concetto Chiave: I Signal Events possono essere combinati e coordinati per creare pattern di comunicazione complessi tra diverse parti di un processo. Questo esempio dimostra come gestire segnali multipli simultanei, segnali in cascata, e pattern avanzati di sincronizzazione.

🎯 Scenario: Sistema di Coordinamento Multi-Segnale

L'esempio illustra un sistema di coordinamento complesso dove diversi segnali vengono emessi e ricevuti in modo orchestrato. Il processo principale gestisce la logica di business mentre tre subprocess separati reagiscono autonomamente ai segnali ricevuti.

📊 Flusso del Processo

1. Sistema si inizializza
2. Emette segnale Alert → Attiva Subprocess 1 (non-interrupting)
3. Attende decisione tramite Event-Based Gateway
4. Riceve Approve o Reject
   
   Path A (Approval):
   → Process Approval → Emette Complete → Attiva Subprocess 2
   
   Path B (Rejection):
   → Process Rejection → Emette Cancel → Attiva Subprocess 3
   → Subprocess 3 emette Alert (cascading) → Ri-attiva Subprocess 1

🔷 I Cinque Segnali del Sistema

Alert

Tipo: Broadcast non-interrupting
Scopo: Notifica generale del sistema per logging e monitoraggio
Gestito da: Subprocess 1 (Event Subprocess con Signal Start)
Caratteristica: Può essere emesso multiplo volte, non interrompe il flusso principale

Approve

Tipo: Decision signal
Scopo: Segnala approvazione di una richiesta
Gestito da: Event-Based Gateway nel processo principale
Caratteristica: Alternativa mutualmente esclusiva con Reject

Reject

Tipo: Decision signal
Scopo: Segnala rigetto di una richiesta
Gestito da: Event-Based Gateway nel processo principale
Caratteristica: Triggera subprocess 3 che genera signal in cascata

Complete

Tipo: Workflow trigger signal
Scopo: Avvia task di lunga durata dopo approvazione
Gestito da: Subprocess 2 (Event Subprocess con Signal Start)
Caratteristica: Può essere interrotto da Cancel via boundary event

Cancel

Tipo: Interrupting signal
Scopo: Interrompe task in esecuzione e attiva cleanup
Gestito da: Boundary Event su Task_LongRunning (interrupting)
Caratteristica: Interrompe immediatamente il task e devia il flusso

🎨 Pattern Dimostrati

📌 Pattern 1: Event-Based Gateway con Signal

Utilizzato per implementare decisioni basate su eventi esterni. Il gateway attende uno tra più signal catch events, permettendo al processo di procedere solo quando uno specifico segnale viene ricevuto.

<eventBasedGateway id="Gateway_EventBased"> <outgoing>Flow_5</outgoing> <!-- to Approve --> <outgoing>Flow_6</outgoing> <!-- to Reject --> </eventBasedGateway> <intermediateCatchEvent id="Event_CatchApprove"> <signalEventDefinition signalRef="Signal_Approve" /> </intermediateCatchEvent> <intermediateCatchEvent id="Event_CatchReject"> <signalEventDefinition signalRef="Signal_Reject" /> </intermediateCatchEvent>
Uso pratico: Sistemi di approvazione, workflow di revisione, pattern di await multipli.

📌 Pattern 2: Event Subprocess con Signal Start

Subprocess triggerati da eventi (triggeredByEvent="true") che si attivano automaticamente quando il segnale viene emesso. Possono essere interrupting o non-interrupting.

<subProcess id="SubProcess_1" triggeredByEvent="true"> <startEvent id="Start_SignalAlert" isInterrupting="false"> <signalEventDefinition signalRef="Signal_Alert" /> </startEvent> <!-- Tasks eseguite in parallelo al processo principale --> </subProcess>
  • isInterrupting="false": esegue in parallelo (Subprocess 1 e 3)
  • isInterrupting="true": interrompe il processo padre (non usato qui)
Uso pratico: Logging parallelo, monitoring, notifiche non bloccanti.

📌 Pattern 3: Signal Boundary Event (Interrupting)

I Boundary Events vengono attaccati a task o subprocess per gestire interruzioni. Quando il segnale viene ricevuto, il task viene interrotto e il flusso segue il boundary event.

<task id="Task_LongRunning" name="Execute Long Task" /> <boundaryEvent id="Boundary_Cancel" attachedToRef="Task_LongRunning" cancelActivity="true"> <signalEventDefinition signalRef="Signal_Cancel" /> </boundaryEvent> <!-- Flow da Boundary_Cancel porta a Cleanup -->
Uso pratico: Timeout management, cancellazione task, gestione errori asincroni.

📌 Pattern 4: Cascading Signals

Un subprocess può emettere nuovi segnali come risultato della ricezione di un altro segnale, creando una catena di reazioni (event chain).

Subprocess 3 (triggeredByEvent="true"): Riceve: Signal_Reject ↓ Esegue: Analyze Rejection ↓ Emette: Signal_Alert (cascading) ↓ Attiva: Subprocess 1 di nuovo (non-interrupting)
Uso pratico: Event chains, notification cascades, pattern di propagazione.

🔍 Analisi dei Subprocess

Subprocess 1: Alert Handler (Non-Interrupting)

Attivazione: Signal Start Event con Signal_Alert (isInterrupting="false")
Comportamento: Esegue in parallelo senza interrompere il flusso principale
Azioni:
  • Log Alert → registra l'evento
  • Notify Team → invia notifiche ai membri del team
Scenario: Può essere attivato più volte (prima dal processo principale, poi dal cascading signal in Subprocess 3)

Subprocess 2: Long Task con Cancellation

Attivazione: Signal Start Event con Signal_Complete
Comportamento: Task di lunga durata con possibilità di interruzione
Azioni:
  • Execute Long Task → operazione time-consuming
  • Cancellation Path: Boundary Event (Cancel) → Cleanup → End (Cancelled)
  • Success Path: Finalize Task → End (Completed)
Scenario: Se arriva Cancel mentre il task è in esecuzione, viene interrotto e viene eseguito cleanup

Subprocess 3: Rejection Handler con Cascading

Attivazione: Signal Start Event con Signal_Reject (isInterrupting="false")
Comportamento: Analizza il rigetto ed emette un segnale secondario
Azioni:
  • Analyze Rejection → analisi del motivo del rigetto
  • Throw Alert (Cascade) → emette nuovo Signal_Alert
  • Create Report → genera documentazione
Scenario: Dimostra come un signal può triggeerare l'emissione di un altro signal (cascading effect)

💡 Concetti Chiave

🔄 Broadcast vs Point-to-Point

I Signal Events sono broadcast: un singolo signal può essere ricevuto da tutti i catch events che lo ascoltano simultaneamente. Nell'esempio:

  • Signal_Alert può attivare Subprocess 1 ogni volta che viene emesso (iniziale + cascading)
  • Signal_Cancel viene ricevuto dal Boundary Event se il task è attivo
  • Multiple listener possono reagire allo stesso signal indipendentemente

⚡ Event-Based Gateway

L'Event-Based Gateway permette decisioni basate su "quale evento arriva per primo". Differenze con Exclusive Gateway:

  • Exclusive Gateway: decisione basata su condizioni dati
  • Event-Based Gateway: decisione basata su eventi temporali

Nel nostro esempio: il processo attende Approve O Reject, e procede con il path corrispondente al primo segnale ricevuto.

🎯 Interrupting vs Non-Interrupting

Differenza cruciale per il comportamento dei subprocess:

Tipo Comportamento Uso nell'esempio
Non-Interrupting Esegue in parallelo, il processo principale continua Subprocess 1 (Alert) e 3 (Reject) - possono essere attivati ripetutamente
Interrupting Interrompe il processo/task corrente Boundary_Cancel su Task_LongRunning - ferma il task immediatamente

🎓 Best Practices

  • Naming convention: Usa nomi descrittivi per i signal (Alert, Approve, Reject vs generici Signal1, Signal2)
  • Event Subprocess: Usa triggeredByEvent="true" per logic che deve reagire a signal ma non è parte del flusso principale
  • Boundary Events: Preferisci cancelActivity="true" per interruzioni definitive, "false" per monitoring
  • Event-Based Gateway: Usa solo con catch events, non con task normali
  • Cascading Signals: Documenta chiaramente le catene di signal per evitare loop infiniti
  • Testing: Testa tutti i path possibili, specialmente le interazioni tra signal multipli simultanei

🚀 Scenari di Utilizzo Reale

1. Sistema di Orchestrazione Microservizi

Scenario: Coordinamento di microservizi asincroni dove il completamento di un servizio triggera altri servizi.
Pattern utilizzati: Event-Based Gateway (attesa risposte), Cascading Signals (propagazione stato), Boundary Events (timeout/cancellazione)

2. Sistema di Approvazione Multi-Livello

Scenario: Workflow che richiede approvazioni sequenziali con possibilità di escalation o cancellazione.
Pattern utilizzati: Event-Based Gateway (attesa decisioni), Non-Interrupting Start (logging/audit paralleli), Interrupting Boundary (cancellazione workflow)

3. Monitoring e Alerting Distribuito

Scenario: Sistema che deve loggare e notificare eventi senza bloccare l'operazione principale.
Pattern utilizzati: Non-Interrupting Event Subprocess (logging parallelo), Cascading Signals (propagazione alerts), Multiple listeners (vari team notificati)

⚠️ Attenzione

Potenziali Problemi

  • Race Conditions: Con Event-Based Gateway, l'ordine di arrivo dei signal può essere non deterministico
  • Signal Storms: Cascading signal possono creare troppi eventi se non controllati
  • Deadlock: If tutti gli event-based gateway attendono signal che nessuno emette
  • Memory Leaks: Subprocess non-interrupting che continuano ad accumularsi se triggerati troppo frequentemente

Soluzioni

  • Implementa timeout sui catch event (usando Timer Event in OR con Signal)
  • Usa correlation keys per distinguere signal destinati a istanze specifiche
  • Implementa rate limiting per cascade signal
  • Monitora il numero di subprocess attivi e implementa cleanup
