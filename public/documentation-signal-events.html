Esempio: Signal Events

Comunicazione broadcast tra processi multipli tramite segnali

← Torna all'indice della documentazione

📡 Obiettivo dell'esempio

Il file signal-events-example.bpmn dimostra come utilizzare i Signal Events in BPMN per coordinare molteplici processi attraverso comunicazioni broadcast. L'esempio modella un sistema di gestione emergenze in un edificio industriale, dove un evento critico deve essere notificato simultaneamente a tutti i reparti.

🔊 Caratteristica chiave: BROADCAST

1 segnale emesso → N processi ricevono

A differenza dei Message Events (punto-a-punto), i Signal Events permettono a un singolo evento di attivare o notificare tutti i processi che lo ascoltano contemporaneamente.

💡 Come utilizzare questo esempio
  1. Carica signal-events-example.bpmn dal menu File
  2. Osserva nel Modeler i 4 pool (Control Center, Sicurezza, Produzione, Admin)
  3. Nota che non ci sono message flow tra i pool (solo sequence flow interni)
  4. Passa alla Simulation e osserva come un singolo segnale attiva tutti i pool

⚡ Signal Events vs Message Events

Differenze fondamentali

📨 Message Events
  • Comunicazione punto-a-punto
  • Un mittente → Un destinatario specifico
  • Richiede message flow esplicito
  • Identifica il processo target
  • Come una lettera postale con indirizzo
Esempio: Cliente invia richiesta → Venditore specifico riceve
📡 Signal Events
  • Comunicazione broadcast
  • Un mittente → Tutti gli ascoltatori
  • Nessun message flow necessario
  • Non identifica destinatari specifici
  • Come una sirena o annuncio pubblico
Esempio: Sistema emette allarme → Tutti i reparti reagiscono
🎯 Quando usare Signal vs Message

Usa Signal Event quando:

  • Devi notificare molteplici processi contemporaneamente
  • Non sai in anticipo quanti processi riceveranno il segnale
  • Vuoi implementare pattern publish-subscribe
  • La comunicazione è unidirezionale (nessuna risposta attesa)

Usa Message Event quando:

  • La comunicazione è tra due processi specifici
  • Serve una conversazione punto-a-punto
  • Il destinatario deve essere identificato esplicitamente
  • Serve tracciare la relazione mittente-destinatario

🏭 Scenario: Sistema di Gestione Emergenze

L'esempio modella un sistema industriale con 4 partecipanti che devono coordinarsi durante situazioni di emergenza:

🎛️ Sistema Centrale

Ruolo: Coordinatore

  • Monitora sensori
  • Emette segnali broadcast
  • Decide se evacuare
  • Risolve l'emergenza
🛡️ Sicurezza

Ruolo: Risposta rapida

  • Attivato da segnale allarme
  • Verifica situazione
  • Guida evacuazione
  • Rapporto finale
⚙️ Produzione

Ruolo: Arresto sicuro

  • Produzione in corso
  • Interrotta da segnale
  • Mette in sicurezza macchinari
  • Riavvia quando possibile
📄 Amministrativo

Ruolo: Gestione persone

  • Lavoro d'ufficio
  • Riceve segnale allarme
  • Salva documenti
  • Evacua se necessario

📡 I 4 segnali broadcast

1. 🚨 Allarme Incendio
  • Emesso da: Sistema Centrale (quando rileva problema)
  • Ricevuto da: Sicurezza (Start Event), Produzione (Boundary), Admin (Boundary)
  • Effetto: Avvia processo Sicurezza, interrompe Produzione e Admin
2. 🏃 Evacuazione
  • Emesso da: Sistema Centrale (se decide evacuazione)
  • Ricevuto da: Sicurezza (Catch Event), Admin (Catch Event)
  • Effetto: Sblocca processi in attesa, avvia evacuazione coordinata
3. ✅ Fine Emergenza
  • Emesso da: Sistema Centrale (quando situazione risolta)
  • Ricevuto da: Sicurezza (Catch), Produzione (Catch), Admin (Catch)
  • Effetto: Tutti i reparti riprendono operazioni normali
4. 🔧 Manutenzione Programmata
  • Emesso da: (non mostrato nell'esempio, ma supportato)
  • Ricevuto da: Produzione (Boundary Event)
  • Effetto: Interrompe produzione per manutenzione pianificata

📋 Tipi di Signal Events

📤Signal Intermediate Throw Event

Simbolo: Cerchio doppio con icona triangolo (vuoto)

Funzione: Emettere un segnale broadcast a tutti gli ascoltatori

Comportamento: Il processo emette il segnale nell'etere. Tutti i processi che hanno un catch/start/boundary event per quel segnale lo ricevono simultaneamente. Il processo mittente continua immediatamente senza aspettare.

Nel diagramma di esempio:
  • Event_ThrowFireAlarm - Sistema Centrale emette "Allarme Incendio"
  • Event_ThrowEvacuation - Sistema Centrale emette "Evacuazione"
  • Event_ThrowAllClear - Sistema Centrale emette "Fine Emergenza"

💡 Nota: Un singolo throw event attiva 3 processi contemporaneamente (Sicurezza, Produzione, Admin).

📥Signal Intermediate Catch Event

Simbolo: Cerchio doppio con icona triangolo (pieno)

Funzione: Attendere la ricezione di un segnale specifico

Comportamento: Il processo si ferma in attesa. Quando qualsiasi processo emette quel segnale (anche uno sconosciuto), il token prosegue. Non c'è legame diretto mittente-destinatario.

Nel diagramma di esempio:
  • Event_CatchEvacuation - Sicurezza aspetta ordine evacuazione
  • Event_CatchAllClear - Sicurezza, Produzione, Admin aspettano fine emergenza
  • Event_AdminEvacuation - Admin aspetta conferma evacuazione

💡 Nota: Questi eventi creano punti di sincronizzazione globale. Quando il Sistema Centrale emette "Fine Emergenza", tutti e 3 i reparti si sbloccano contemporaneamente.

▶️Signal Start Event

Simbolo: Cerchio singolo con icona triangolo (pieno)

Funzione: Avviare un processo quando viene emesso un segnale specifico

Comportamento: Il processo non esiste fino a quando qualcuno emette il segnale. Quando il segnale arriva, una nuova istanza del processo viene creata. Possono essere create molteplici istanze se il segnale viene emesso più volte.

Nel diagramma di esempio:
  • Start_Security_Signal - Il processo Sicurezza si avvia solo quando riceve il segnale "Allarme Incendio"

💡 Caso d'uso: Ideale per processi "on-demand" o "event-driven" che devono attivarsi in risposta a eventi globali. Il Dipartimento Sicurezza non è sempre attivo, ma viene "svegliato" dall'allarme.

⚠️Signal Boundary Event

Simbolo: Cerchio doppio con triangolo, attaccato al bordo di un'attività

Funzione: Interrompere un'attività in corso quando viene emesso un segnale

Comportamento: Mentre l'attività è in esecuzione, se qualsiasi processo emette il segnale specificato (anche uno non collegato direttamente), l'attività viene interrotta e il flusso segue il percorso del boundary event.

Nel diagramma di esempio:
  • Boundary_FireAlarm - Attaccato a "Produzione in Corso" nel reparto Produzione
  • Boundary_Maintenance - Attaccato a "Produzione in Corso" (secondo boundary)
  • Boundary_AdminAlarm - Attaccato a "Lavoro d'Ufficio" nel reparto Admin

💡 Potenza: Un singolo segnale "Allarme Incendio" interrompe simultaneamente sia la Produzione che l'Amministrazione, anche se non sanno chi ha emesso il segnale.

🎮 Guida alla simulazione

Fase 1: Avvio e monitoraggio normale

Cosa succede:
  1. Sistema Centrale: Inizia con Start Event, esegue "Monitora Sensori"
  2. Produzione: Inizia con Start Event, esegue "Produzione in Corso"
  3. Admin: Inizia con Start Event, esegue "Lavoro d'Ufficio"
  4. Sicurezza: NON inizia (aspetta segnale di allarme)

Osserva: Sicurezza è l'unico processo che non parte automaticamente. Ha un Signal Start Event che lo mantiene dormiente.

Fase 2: Rilevazione emergenza

Sistema Centrale raggiunge Gateway "Rilevato problema?"

Se seleziona "Incendio":

  1. Arriva a Event_ThrowFireAlarm (Signal Throw Event)
  2. 🔊 BROADCAST: Segnale "Allarme Incendio" viene emesso nell'etere
  3. ⚡ Effetto simultaneo su 3 pool:
    • Sicurezza: Il processo SI AVVIA da Start_Security_Signal
    • Produzione: "Produzione in Corso" viene INTERROTTA da Boundary_FireAlarm
    • Admin: "Lavoro d'Ufficio" viene INTERROTTO da Boundary_AdminAlarm

💡 Nota critica: Un singolo throw event ha attivato/interrotto 3 processi diversi contemporaneamente. Questa è la potenza del broadcast signal!

Fase 3: Evacuazione condizionale

Sistema Centrale valuta situazione:
  1. Esegue "Valuta Situazione"
  2. Raggiunge Gateway "Evacuare?"
  3. Se sceglie "Sì":
    • Emette segnale Event_ThrowEvacuation
    • Sicurezza: Si sblocca da Event_CatchEvacuation, esegue "Guida Evacuazione"
    • Admin: Si sblocca da Event_AdminEvacuation, esegue "Evacua Ufficio"
  4. Se sceglie "No":
    • Sicurezza rimane bloccata (attende evacuazione che non arriva mai - caso teorico)
    • Sistema coordina risposta senza evacuazione totale

Fase 4: Risoluzione e ripristino

Sistema Centrale risolve emergenza:
  1. Esegue "Risolvi Emergenza"
  2. Emette segnale Event_ThrowAllClear
  3. 🔊 BROADCAST "Fine Emergenza":
    • Sicurezza: Si sblocca da Event_CatchAllClear, fa rapporto finale
    • Produzione: Si sblocca da Event_ProdAllClear, riavvia produzione
    • Admin: Si sblocca da Event_AdminAllClear, riprende lavoro
  4. Tutti i processi ritornano a operazioni normali e terminano

⚡ Sincronizzazione globale: Di nuovo, un singolo segnale ha sbloccato contemporaneamente 3 processi che erano tutti in attesa. Tutti riprendono "in sincronia".

🎯 Suggerimenti per la simulazione
  • Velocità lenta: Imposta velocità molto bassa per osservare l'effetto broadcast
  • Pausa strategica: Metti in pausa PRIMA di ogni Throw Event per anticipare l'effetto
  • Osserva i 4 pool: Assicurati che tutti i pool siano visibili contemporaneamente
  • Conta i token: Quando viene emesso un segnale, conta quanti token si muovono simultaneamente
  • Testa percorsi diversi: Prova "Tutto OK" per vedere il flusso normale senza emergenza
  • Log eventi: Il log mostrerà che eventi in pool diversi accadono nello stesso istante

💼 Esempio pratico: Trading System

Scenario: Sistema di trading con molteplici strategie automatiche

Signal "Market Volatility": Quando il sistema rileva alta volatilità, emette un segnale che:

  • Strategy A: Riduce posizioni aperte (Boundary Event interrompe trading normale)
  • Strategy B: Attiva hedging (Catch Event sblocca procedura protezione)
  • Strategy C: Invia notifiche ai trader (Start Event avvia processo notifica)
  • Risk Management: Aumenta margini richiesti (Catch Event attiva procedure rischio)

Vantaggio: Tutte le strategie reagiscono simultaneamente allo stesso evento di mercato senza dover comunicare punto-a-punto.

🔄 Pattern avanzati con Signal Events

Pattern 1: Multiple Instances (Istanze Multiple)

Se più processi identici hanno lo stesso Signal Start Event, emettere il segnale crea istanze multiple del processo.

Esempio: 10 operatori hanno processi con Signal Start "Chiamata Cliente Urgente". Quando arriva una chiamata urgente, il segnale può attivare tutti e 10 gli operatori contemporaneamente, e il primo che risponde la prende.

Pattern 2: Cascading Signals (Segnali a Cascata)

Un processo che riceve un segnale può emettere a sua volta un altro segnale, creando una catena di reazioni.

Esempio: Segnale "Sistema Offline" → Process Monitor riceve → Emette "Failover" → Backup Systems ricevono → Ognuno emette "Service Ready" → Load Balancer riconfigura

Pattern 3: Optional Reception (Ricezione Opzionale)

Non tutti i processi devono ricevere ogni segnale. Solo chi ha definito un listener (catch/start/boundary) per quel segnale viene notificato.

Nell'esempio: Il segnale "Manutenzione Programmata" viene ricevuto solo da Produzione (ha il Boundary Event), mentre Sicurezza e Admin lo ignorano (non hanno listener).

✅ Best Practices

Quando usare Signal Events

  • ✅ Notifiche broadcast: Devi informare tutti i processi simultaneamente
  • ✅ Eventi globali: Qualcosa accade che interessa molteplici parti del sistema
  • ✅ Coordinazione loosely-coupled: I processi non devono conoscersi a vicenda
  • ✅ Event-driven systems: Architetture reattive basate su eventi
  • ✅ Sincronizzazione globale: Tutti devono fermarsi/ripartire insieme

⚠️ Attenzione a...

  • ❌ Non usare per comunicazioni specifiche: Se devi inviare a UN destinatario preciso, usa Message
  • ❌ Non aspettarti risposte: I Signal sono unidirezionali, non c'è acknowledgment
  • ❌ Naming chiaro: Usa nomi descrittivi per i segnali (non "Signal1", "Signal2")
  • ❌ Gestisci assenza di listener: Un segnale emesso senza listener è perso (non è un errore)
  • ❌ Ordine non garantito: Se emetti più segnali velocemente, l'ordine di ricezione può variare

🎓 Linee guida di design

  1. Nomina segnali in modo esplicito: "EmergencyEvacuation" non "Signal1"
  2. Documenta chi emette e chi riceve: Mantieni una tabella dei segnali e listener
  3. Usa Signal Definitions: Definisci segnali a livello di diagram per riuso
  4. Considera timeout: Aggiungi Timer Boundary Events per gestire segnali che non arrivano
  5. Testa scenari senza listener: Verifica che emettere segnali "nel vuoto" non causi problemi
